# 첫 한 달 운영 플랜 — 아크 + av/rf MAB A/B (2026-06-07 수립)

> 목적: ① 응집력 있는 첫 달 스토리(아크)를 굴리면서 ② **av vs rf 중 무엇이 더 잘 되는지 multi-armed bandit으로 검증**한다. 업로드 피드백(YouTube Analytics)을 보상으로 사용.

## 1. 시즌 스토리 아크 (아크 LLM 생성, **실제 클립 인벤토리에 그라운딩**)
**테마: "여름의 소리와 파도 퀘스트 — 현실 산책과 상상의 바다가 이어지는 6월"**
> ⚠️ rf 비트는 **실제 보유 footage에서만** (PD 2026-06-07). 아카이브엔 수박/실제 수영 클립이 **없음** → 그건 av 상상 전용. rf는 인벤토리에 풍부한 것으로: outdoor 산책 92·home 105·카페·먹방·장난감·낮잠·그루밍·하네스 외출.
- **1주차** 캐릭터 소개 + 여름 퀘스트 훅('빛나는 소라'). av=밤 해변·바다⇄우주 전환 / **rf=그늘 산책(하네스/리드), 창가 낮잠(쿠션/담요), 카페 앞 쉬기**.
- **2주차** 관계 강화 + **월간 랴니&레오 재소개 스페셜**(av 배경전환: 수영장→장마구름→야시장→궤도) / **rf=장마 전 점검·실내 포커스 훈련(트릿), 수분 루틴(물그릇), 골든아워 산책**.
- **3주차** 성장·첫 고비(장마): **rf=비 오는 날 DIL(창가 구경+실내 놀이), 밸런스 놀이(쿠션/스크래처/스틱), 그루밍 쿨다운** / av=장마 스톰 던전·첫 미니 서핑.
- **4주차** 떡밥 회수·피날레: av=빛의 파도 대회(랴니 공포 극복)·7월 티저 / **rf=노을 모먼트+6월 하이라이트, 그늘 피크닉 톤 산책, '여름 사운드 팩'(우리가 담은 소리)**.
- 열린 떡밥(7월 이월): 소라의 기원, 랴니 현실 물놀이(footage 생기면), '여름의 소리' 팬참여, 라이벌 서퍼 NPC.
- **자동 그라운딩**: `arc._asset_inventory()`가 실제 활동/장소/소품 통계를 플래너 LLM에 먹이고, "rf=인벤토리 내에서만, av=상상 가능" 규칙 강제. → 앞으로 플랜이 없는 소재를 rf에 안 넣음.

## 1b. 채널 첫 오픈 — 캐릭터 소개 + 과거⇄현재 클립 (PD 2026-06-07)
**첫 달은 채널의 첫 오픈**이라 새 시청자에게 두 주인공을 각인시켜야 한다.
- **캐릭터 소개를 1주차 핵심**으로(랴니=11살 의젓한 누나/엄마, 레오=8개월 장난꾸러기 아들). 1주차+2주차 재소개 스페셜로 정체성 확립.
- **클립 소싱 = 과거(아카이브) ⇄ 현재 왕복**: 최신 클립만 쓰지 말고, 입양 첫날·아기 때·옛 추억 같은 **과거 아카이브 클립**을 섞어 캐릭터 서사를 깊게.
  - 예: "랴니가 우리집에 온 날(N년 전)" 아카이브 → "지금의 랴니" 현재. 레오 아기→현재.
- **★ 시간 맥락을 캡션이 반드시 설명**(메모리레인 톤): 과거 클립엔 "○년 전 아기 랴니", "입양 첫날", "그때는…", 현재 클립엔 "지금은…", "여전히…". 설명 없이 과거/현재를 섞으면 시청자가 혼란 → 갑툭튀 금지.
  - DB의 `captured_iso`/`milestones`(입양기념일·생일)로 실제 시점 근거. 추측 금지(실제 날짜 기반).
- 구현 훅:
  - producer `available_videos`가 **최근 20개만** 주므로, 소개/메모리 회차엔 **아카이브 클립도 후보로** 공급(예: 연도별 샘플링 or memory-lane 쿼리). 현재는 `ORDER BY captured_iso DESC LIMIT 20` → 과거 미포함.
  - 캡션 프롬프트(realfootage_concept.md/caption_agent.md)에 **"과거 클립이면 시점 명시"** 규칙 추가.
  - `milestones`(입양/생일) 연동해 메모리레인 회차 트리거.

## 2. av vs rf — Multi-Armed Bandit A/B 설계
**Arms(팔):** `ai_vtuber`, `real_footage`.
**보상(reward):** 업로드 ~48–72h 후 YouTube Analytics에서 정규화한 참여도.
- 1차 지표: **48h 조회수**(Shorts는 노출-조회 주도) + **평균 시청 지속율(retention %)**.
- 합성 reward = z-정규화(views_48h) 0.6 + retention% 0.4 (가중치 튜너블). 0~1로 squash.
**알고리즘:** **Thompson Sampling**(팔별 가우시안 posterior). 저volume·지연보상에 적합, 불확실성 기반 자연 탐색.
- **Week 1 = 워밍업(강제 50/50)**: 두 팔에 동등 데이터 시드. 이후 posterior로 할당.
- 완전 착취 금지(Thompson은 항상 약간 탐색) → 한쪽이 이겨도 소수 비율 유지.
**할당 방식:** 첫 달은 **매일 av·rf 둘 다 생산(paired)** 해서 비교 데이터 최대 수집. MAB는 "다음 달 데일리 비율"을 결정(예: av 우세면 2 av:1 rf). 즉 첫 달=실험+데이터, 결론은 월말.

### 교란변수 통제 (PD 강조: 시간·소재·스토리 잘 선정)
- **소재(topic) 페어링:** 같은 주차 테마를 av·rf 둘 다에 부여(예: 1주차=수박/여름). 측정되는 차이는 **포맷(레인)**이지 소재가 아니게. (아크가 이미 주차 테마 공유 → 유지.)
- **업로드 시각 고정(1차):** av-vs-rf 실험 동안 **시각은 고정**(혼동 방지). 시각 A/B는 레인 결론 후 2차 실험으로 분리. (저volume에 동시 2요인 실험 금지.)
  - 단, 슬롯이 여러 개면 av·rf에 **시각을 카운터밸런스**(같은 시각 분포)로 배정.
- **요일:** 두 레인을 같은 요일 분포로(주말 효과 분리).

## 3. 업로드 시각 (1차 고정값 + 2차 실험 후보)
- **1차 고정:** 저녁 프라임 **20:00 KST** (Shorts 저녁대 강세). `YOUTUBE_PUBLISH_HOUR=20`.
  - 둘 다 생산 시: av=20:00, rf=21:00(또는 익일 분산) — 같은 분포로 카운터밸런스.
- **2차(레인 결정 후) 시각 A/B 후보:** 08:00(출근) / 12:30(점심) / 20:00(저녁). MAB 동일 로직 재사용.

## 4. 측정·결정 규칙
- 각 영상 publish 후 ~48h에 Analytics 수집 → `video_performance`에 reward 기록 → 밴딧 posterior 업데이트.
- **결정:** 각 팔 N≥8~10 누적 + 한 팔이 90%+ 확률로 우세하면 다음 달 비율 이동. 아니면 첫 달 동안 50/50 유지하며 계속 수집.
- 보고: 주 1회 슬랙에 "av vs rf 누적 reward·표본수·현재 우세확률".

## 5. 구현 로드맵 (이 플랜을 돌리려면)
1. **`youtube/analytics.py`**: `get_analytics()`(있음)로 video별 views/retention 조회. publish 후 48h 잡으로 수집.
2. **`video_performance` 테이블**: video_id, lane, views_48h, retention, reward, fetched_at.
3. **`agents/bandit.py`**: 팔별 posterior(Thompson) 업데이트 + 다음 레인/비율 선택. 보상 정규화 포함.
4. **스케줄 잡**: publish+48h에 analytics 수집 → reward → bandit 업데이트(launchd/cron).
5. **producer 연동:** 유연-레인 슬롯에서 bandit.choose() 참조(워밍업 동안은 50/50 강제).
6. **`ARC_ENABLED=1`**: 첫 달 운영 시작 시 켠다(아크가 시즌 플랜·디렉티브·기록 구동).

## 6. 리스크/주의
- **지연보상**: Shorts 조회는 며칠에 걸쳐 누적 → 48h 스냅은 근사치. 7d 보정 옵션 둘 것.
- **저volume**: 표본 적어 결론 느림 → paired 생산으로 표본↑, Thompson으로 조기 착취 방지.
- **교란**: 시각/소재 통제 안 하면 레인 비교 무의미 → §2 통제 필수.
- **품질 게이트 우선**: 기리 통과한 것만 업로드(저품질이 reward 오염 안 되게).
