# 첫 한 달 운영 플랜 — 아크 + av/rf MAB A/B (2026-06-07 수립)

> 목적: ① 응집력 있는 첫 달 스토리(아크)를 굴리면서 ② **av vs rf 중 무엇이 더 잘 되는지 multi-armed bandit으로 검증**한다. 업로드 피드백(YouTube Analytics)을 보상으로 사용.

## 0. 전제 (지금 갖춰진 것)
- YouTube 업로드 파이프라인 완성(OAuth·예약공개·uploaded=1·video_id 기록). 채널 Ryani_n_Leo.
- 아크 시스템(`agents/arc.py`): 통합 대장 + 롤링 시즌 플랜(계절/공휴일/트렌드/월1회 재소개/판타지). `ARC_ENABLED`로 on.
- 클립 4편 쿨다운(업로드된 것만 카운트).

## 1. 시즌 스토리 아크 (아크 LLM 생성, **실제 클립 인벤토리에 그라운딩**)
**테마: "여름의 소리와 파도 퀘스트 — 현실 산책과 상상의 바다가 이어지는 6월"**
> ⚠️ rf 비트는 **실제 보유 footage에서만** (PD 2026-06-07). 아카이브엔 수박/실제 수영 클립이 **없음** → 그건 av 상상 전용. rf는 인벤토리에 풍부한 것으로: outdoor 산책 92·home 105·카페·먹방·장난감·낮잠·그루밍·하네스 외출.
- **1주차** 캐릭터 소개 + 여름 퀘스트 훅('빛나는 소라'). av=밤 해변·바다⇄우주 전환 / **rf=그늘 산책(하네스/리드), 창가 낮잠(쿠션/담요), 카페 앞 쉬기**.
- **2주차** 관계 강화 + **월간 랴니&레오 재소개 스페셜**(av 배경전환: 수영장→장마구름→야시장→궤도) / **rf=장마 전 점검·실내 포커스 훈련(트릿), 수분 루틴(물그릇), 골든아워 산책**.
- **3주차** 성장·첫 고비(장마): **rf=비 오는 날 DIL(창가 구경+실내 놀이), 밸런스 놀이(쿠션/스크래처/스틱), 그루밍 쿨다운** / av=장마 스톰 던전·첫 미니 서핑.
- **4주차** 떡밥 회수·피날레: av=빛의 파도 대회(랴니 공포 극복)·7월 티저 / **rf=노을 모먼트+6월 하이라이트, 그늘 피크닉 톤 산책, '여름 사운드 팩'(우리가 담은 소리)**.
- 열린 떡밥(7월 이월): 소라의 기원, 랴니 현실 물놀이(footage 생기면), '여름의 소리' 팬참여, 라이벌 서퍼 NPC.
- **자동 그라운딩**: `arc._asset_inventory()`가 실제 활동/장소/소품 통계를 플래너 LLM에 먹이고, "rf=인벤토리 내에서만, av=상상 가능" 규칙 강제. → 앞으로 플랜이 없는 소재를 rf에 안 넣음.

## 1b. 채널 첫 오픈 — 캐릭터 소개 + 과거⇄현재 클립 (PD 2026-06-07)
**첫 달은 채널의 첫 오픈**이라 새 시청자에게 두 주인공을 각인시켜야 한다.
- **캐릭터 소개를 1주차 핵심**으로(랴니=11살 의젓한 누나/엄마, 레오=8개월 장난꾸러기 아들). 1주차+2주차 재소개 스페셜로 정체성 확립.
- **클립 소싱 = 과거(아카이브) ⇄ 현재 왕복**: 최신 클립만 쓰지 말고, 입양 첫날·아기 때·옛 추억 같은 **과거 아카이브 클립**을 섞어 캐릭터 서사를 깊게.
  - 예: "랴니가 우리집에 온 날(N년 전)" 아카이브 → "지금의 랴니" 현재. 레오 아기→현재.
- **★ 시간 맥락을 캡션이 반드시 설명**(메모리레인 톤): 과거 클립엔 "○년 전 아기 랴니", "입양 첫날", "그때는…", 현재 클립엔 "지금은…", "여전히…". 설명 없이 과거/현재를 섞으면 시청자가 혼란 → 갑툭튀 금지.
  - DB의 `captured_iso`/`milestones`(입양기념일·생일)로 실제 시점 근거. 추측 금지(실제 날짜 기반).
- 구현 훅:
  - producer `available_videos`가 **최근 20개만** 주므로, 소개/메모리 회차엔 **아카이브 클립도 후보로** 공급(예: 연도별 샘플링 or memory-lane 쿼리). 현재는 `ORDER BY captured_iso DESC LIMIT 20` → 과거 미포함.
  - 캡션 프롬프트(realfootage_concept.md/caption_agent.md)에 **"과거 클립이면 시점 명시"** 규칙 추가.
  - `milestones`(입양/생일) 연동해 메모리레인 회차 트리거.

## 2. 실험 설계 — explore-heavy 4편/day 요인실험 (PD 2026-06-07 확정)
> PD 결정: 런칭 한 달은 **하루 4편**으로 빠르게 explore하고, 월말에 **잘 나온 방식(들)을
> winner로 선정** → 다음 달 exploit. 단순 av-vs-rf 반복이 아니라 **요인실험**으로 슬롯을 쓴다.

**요인(factors):**
- **레인(lane):** `ai_vtuber` / `real_footage` — 1차 관심사.
- **시각(timeslot):** 4개 (§3) — 동시에 최적 시각도 학습.
- (확장 여지) 편집 컨셉·소재 톤 — 저volume이라 1차엔 레인×시각 2요인만. 표본 쌓이면 추가.

**설계 = 레인×시각 라틴스퀘어 (counterbalanced):**
- 하루 4슬롯 = **2 av + 2 rf** 고정(매일 50/50 자동 충족).
- 어느 레인이 어느 시각에 가는지를 **매일 회전**해 레인·시각 교란 제거.
  - Day1: 08=av, 12:30=rf, 18=av, 21=rf
  - Day2: 08=rf, 12:30=av, 18=rf, 21=av  → 2일마다 각 레인이 각 시각에 1회씩.
- **Cannibalization:** 같은 날 4편이 서로 잡아먹어 절대 조회수는 눌리나, **모두 똑같이 눌리므로
  상대 순위(누가 이기나)는 보존** → winner 선정은 상대 비교라 4편/day가 오히려 유리.

**알고리즘:** **Thompson Sampling** — arm = 레인×시각 조합(2×4=8). 요인실험이라 **marginal 효과
(레인만 / 시각만)도 pooling으로 추정** → "av가 낫다", "18시가 낫다" 같은 결론을 별도로 뽑음.
- **Week 1 = 강제 균형(라틴스퀘어 그대로, 탐색 100%)**: 모든 arm에 동등 시드.
- 이후도 런칭 달은 explore 우선(2 av:2 rf 유지). Thompson은 **다음 달 비율 결정용**으로 누적.

### 교란변수 통제
- **소재(topic) 페어링:** 같은 주차 테마를 av·rf 둘 다에(아크가 주차 테마 공유 → 유지).
- **시각:** 라틴스퀘어로 레인×시각 균형 → 시각이 레인 비교를 오염시키지 않음.
- **요일:** 매일 4편이라 요일 효과는 양 레인에 동일 분포(자동 통제).
- **품질:** 아래 §품질 바닥선 — junk가 reward 오염 못 하게.

### ★ 품질 바닥선 (양보 불가)
- **4편은 목표, 품질은 바닥선.** 기리 통과 못 한 영상은 슬롯을 비워(그날 3편) — junk로 채우지 않음.
  저품질 영상의 저조한 성과는 "레인 탓"이 아니라 "그 영상 탓"이라 데이터를 오염시킨다.

## 2b. 검수 워크플로 — 기리 게이트 + PD 스팟체크 (PD 2026-06-07 확정)
> 하루 4편은 회차별 수동 검수+수정이 불가 → 워크플로 전환.
- 생성 → **기리 통과 = 자동 예약업로드**(블로킹 PD 승인 폐지, 런칭 모드 한정).
- 4편 모두 **슬랙 스레드에 결과 mp4 포스팅** → PD가 빠르게 훑고 **이상한 것만 veto/내림**.
- 필요한 신규 훅: `/veto <video_id>` (또는 카드) = YouTube 비공개/삭제 + 카드 uploaded 회수.
- 런칭 모드 플래그(예: `LAUNCH_MODE=1` / `AUTO_PUBLISH_ON_GIRI=1`)로 기존 승인-대기 루프와 분리.

## 3. 업로드 시각 — 4슬롯 (라틴스퀘어 회전)
- **4개 시각(KST, 1차안):** **08:00**(출근) / **12:30**(점심) / **18:00**(퇴근) / **21:00**(저녁 프라임).
- 매일 레인을 시각에 라틴스퀘어로 배정(§2). → 레인 승자 + 최적 시각 동시 도출.
- `YOUTUBE_PUBLISH_HOUR` 단일값 대신 **시각 리스트** 필요(스케줄러가 4슬롯 예약).

## 4. 측정·결정 규칙
- 각 영상 publish 후 ~48h에 Analytics 수집 → `video_performance`에 reward 기록 → posterior 업데이트.
- **보상:** z-정규화(views_48h) 0.6 + retention% 0.4 (튜너블), 0~1 squash. 7d 보정 옵션.
- **결정(월말):** ① **레인 marginal** — 한 레인이 90%+ 우세 → 다음 달 비율 이동(예 2:1). ②
  **시각 marginal** — 최적 시각대(들) 채택. ③ 표본 부족하면 다음 달도 explore 연장(Thompson).
- 보고: **주 1회 슬랙** — 레인별/시각별 누적 reward·표본수·현재 우세확률.

## 5. 구현 로드맵 (이 플랜을 돌리려면)
> 상태: 아크·업로드·쿨다운 ✅ / bandit·analytics·video_performance·아카이브소싱·런칭스케줄러 ❌
1. **`ARC_ENABLED=1`** + 4슬롯 **런칭 스케줄러**: 매일 라틴스퀘어로 (레인, 시각) 4건 생성·예약.
2. **검수 전환**: 기리 통과 자동 예약업로드 + 슬랙 4편 포스팅 + `/veto` 커맨드(런칭 모드 플래그).
3. **아카이브 소싱**: producer가 최근 20개 외 연도별/마일스톤(입양·생일) 클립도 후보 공급 +
   캡션 프롬프트 "과거 클립이면 시점 명시" 규칙. (1주차 캐릭터 소개 = 과거⇄현재 필수)
4. **측정 루프**(첫 영상 publish+48h 전까지): `youtube/analytics.py` + `video_performance` 테이블 +
   `agents/bandit.py`(요인 posterior + marginal) + publish+48h 수집 잡(launchd).
   - Week1은 라틴스퀘어 강제라 bandit 없이도 런칭 가능 → 측정 루프는 병렬로.
5. **첫 회차 = rf 실제 클립 소개**(과거⇄현재 메모리레인)부터. cold-start엔 진정성 훅이 강함.

## 6. 리스크/주의
- **Throughput/비용**: 4편×30일=~120편. av Seedance ~$3/편(≈60편→~$180)+Gemini. 비용 OK,
  **진짜 병목은 품질 유지 생산량** → §품질 바닥선으로 방어(4 목표·품질 바닥).
- **지연보상**: Shorts 조회는 며칠 누적 → 48h 스냅은 근사치. 7d 보정 옵션.
- **저volume**: 4편/day로 표본↑(월 ~120) → 요인실험이라 marginal로 결론 가속. 그래도 부족하면 연장.
- **교란**: 라틴스퀘어로 레인×시각 통제(§2). 안 하면 비교 무의미.
- **품질 게이트 우선**: 기리 통과한 것만 업로드(저품질이 reward·채널 첫인상 오염 안 되게).
- **rf 에셋 런웨이**: 60 rf편 + 4편 쿨다운이 아카이브를 빨리 소진할 수 있음 → 인벤토리(outdoor 92·
  home 105 등) 충분 추정이나 런칭 전 재확인 권장.
